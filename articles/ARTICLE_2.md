# How Hyperpush + Bags.fm turns error tracking into an OSS revenue engine.

Open-source maintainers don't have a talent problem. They have a money problem.

The code gets written. The bugs get filed. The PRs sit open. Not because nobody knows how to fix them — because nobody's getting paid to. The people who maintain the software the internet runs on do it between jobs, after hours, on goodwill that burns out.

We've seen every version of the "fund open source" pitch. Donations. Sponsorship tiers. License games. They all share the same flaw: they depend on someone deciding to give money away. That's charity. Charity doesn't scale.

What scales is a system where money moves because the system works — where funding is a byproduct of usage, not a favor.

That's what we're building.

## The integration: Hyperpush × Bags.fm

When an open-source project signs up for Hyperpush — our open-source, self-hostable Sentry replacement — the onboarding flow now includes a single new step:

**A project token is created through the Bags.fm API.**

Bags.fm is a Solana token launchpad with a full REST API and TypeScript SDK. It handles token creation, trading, fee distribution, and automatic holder dividends — all on-chain, all programmatic.

Here's the actual flow:

1. **Project connects to Hyperpush.** Installs the SDK, starts tracking errors. Standard error tracking setup — stack traces, issue grouping, alerts, the works.

2. **Hyperpush calls Bags.fm's `createTokenInfoAndMetadata` endpoint.** A project token is minted on Solana. The project doesn't need to touch a wallet, understand SPL tokens, or write a single line of Solana code. It happens behind the scenes during setup.

3. **Fee shares are configured via `createFeeShareConfig`.** This is where it gets interesting. Bags.fm lets you split trading fees between multiple wallets at the protocol level. Hyperpush configures a split:
   - A percentage flows to the **project treasury** — funding bounties, development, and maintenance.
   - A percentage flows to a **developer reward pool** — paying the people who actually fix the bugs.
   - A percentage flows to **Hyperpush** — sustaining the platform itself.

4. **The token is live.** Tradeable on Solana. Every trade generates fees. Those fees split automatically across the configured wallets. No invoices. No payment processors. No human in the loop.

That's it. The project now has a funding mechanism that's directly connected to its error tracking infrastructure.

## What Bags.fm actually does under the hood

This isn't a vague "blockchain integration." Bags.fm provides the concrete financial plumbing:

**Token creation** — `sdk.tokenLaunch.createTokenInfo()` mints a new SPL token with metadata, image, social links, and an optional initial buy. The project token exists on-chain in one API call.

**Fee share configuration** — `sdk.config.createFeeShareConfig()` sets up a protocol-level fee split between wallets. By default, partner keys receive 25% (2,500 bps) of fees generated by tokens launched through them. Hyperpush's partner key means every project token automatically funds the ecosystem.

**Trading** — `sdk.trade.getQuote()` and `sdk.trade.createSwapTransaction()` enable trading across Solana DEXs and liquidity pools. Anyone can buy or sell the project token. Every trade generates fees.

**Automatic dividends** — When a token accumulates ≥10 SOL in unclaimed earnings, Bags.fm distributes proportional payouts to the top 100 holders every 24 hours. Directly into wallets. No claiming, no gas games.

**Lifetime fee tracking** — `GET /api/v1/token-launch/lifetime-fees?tokenMint={mint}` gives real-time visibility into total fees collected by any token. Projects can see exactly how much their token has generated.

The mechanical result: every project on Hyperpush has a tradeable token. Every trade funds the project. Every fix funded by the project pays a developer. The financial loop closes without anyone writing a check.

## How OSS developers actually get paid

The public bug board is where this becomes tangible for individual developers.

Hyperpush surfaces tracked errors publicly (opt-in) with bounties attached. The bounties are funded from the project treasury — which is funded by token trading fees. The cycle:

1. **Bugs surface on the public board.** Real errors from real production environments, with stack traces, context, and severity.

2. **Developers pick a bug.** They can see the bounty amount, the project, the severity, and the relevant code context.

3. **Developer submits a PR.** The project maintainers review and verify the fix.

4. **Bounty is distributed.** Hyperpush triggers the payout — in the project's token, in SOL via the token's trading pair, or through the Bags.fm fee distribution. The developer gets paid. No invoices. No 30-day net terms. No chasing anyone.

For developers who hold project tokens they've earned, the automatic dividend system means they continue earning from the project's success long after the fix ships. Fix a critical bug, hold the tokens, and the project's continued usage generates ongoing returns.

## What this is not

This is not a pitch to slap a token on a GitHub repo and pray.

The error tracker has to work. If Hyperpush isn't a genuinely good Sentry replacement without the token layer, the rest is decoration. Stack traces, issue grouping, alerts, performance monitoring, AI root-cause analysis, GitHub integration — all of that has to be solid before the economics matter.

The token isn't the product. The token is the funding mechanism that makes the product sustainable for the people who use it and the people who improve it.

Bags.fm isn't the product either. It's the infrastructure — the Solana-native launchpad that handles the hard parts of token creation, trading, fee distribution, and dividends so Hyperpush doesn't have to build a financial system from scratch.

The product is an error tracker that funds its own ecosystem.
